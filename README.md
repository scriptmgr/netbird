# NetBird Self-Hosted Installer

This repo provides a single self-contained `install.sh` that bootstraps and updates a self-hosted NetBird stack for production-oriented use.

It installs and manages:

- Keycloak (IdP — OIDC, user management, admin UI)
- NetBird management
- NetBird dashboard
- NetBird signal
- Coturn

All templates and logic are baked into `install.sh`. At install time, the script writes runtime-owned files under `/opt/netbird` and preserves generated secrets across reruns.

## What the script does

- Installs Docker Engine and the Compose plugin from the official Docker repository
- Creates a persistent runtime layout under `/opt/netbird`
- Generates and preserves secrets for:
  - Keycloak admin account
  - Keycloak database access
  - Coturn
  - NetBird datastore encryption
- Configures SELinux-friendly bind mounts when SELinux is enabled
- Opens the required firewalld ports on AlmaLinux/RHEL-family hosts
- Publishes localhost-only backends for an external reverse proxy
- Auto-configures Keycloak (realm, PKCE client, service account) on first run
- Safely reruns without rotating generated credentials unless you replace files under `/opt/netbird/secrets`

## Runtime layout

The installer writes the following runtime assets:

- `/opt/netbird/compose/docker-compose.yml`
- `/opt/netbird/etc/keycloak.env`
- `/opt/netbird/etc/netbird.env`
- `/opt/netbird/etc/dashboard.env`
- `/opt/netbird/etc/management.json`
- `/opt/netbird/etc/turnserver.conf`
- `/opt/netbird/secrets/*`

These files are generated by the script; they are not repo-side dependencies.

## Requirements

- Linux host
- Root access
- Internet access to Docker Hub, Quay.io, and GHCR
- A public DNS name for the deployment
- An external reverse proxy you manage separately

## Installation

```sh
git clone https://github.com/scriptmgr/netbird
cd netbird
sudo sh ./install.sh
```

## Local backends for your reverse proxy

The script binds service backends to localhost so a host reverse proxy can publish them externally:

| Purpose | Local backend |
| --- | --- |
| Dashboard | `http://127.0.0.1:18080` |
| Management HTTP / websocket proxy | `http://127.0.0.1:18081` |
| Keycloak UI / OIDC | `http://127.0.0.1:18082` |
| Signal | `http://127.0.0.1:10000` |
| TURN | `udp/3478`, `udp/49152-49252` |

## Reverse proxy expectations

The reverse proxy is intentionally not provisioned by the installer. It should expose your public domain on the external port you choose and route NetBird paths to the localhost backends above.

Typical path split:

- Dashboard: `/*` → dashboard backend
- Management: `/api/*`, `/ws-proxy/management*`, `/management.ManagementService/*` → management backend
- Signal: `/ws-proxy/signal*`, `/signalexchange.SignalExchange/*` → signal backend
- Keycloak: `/realms/*`, `/resources/*`, `/admin/*` → Keycloak backend

## AlmaLinux 9 / RHEL-family behavior

On AlmaLinux 9 and similar hosts, the script also:

- installs `firewalld`, `policycoreutils-python-utils`, and `container-selinux`
- enables `firewalld`
- opens:
  - external TCP port for the reverse proxy
  - `3478/udp`
  - `49152-49252/udp`

## Keycloak admin UI

Keycloak is fully managed through its web UI. After install the admin console is available at:

```
https://<your-domain>:<port>
```

- **Username:** `admin`
- **Password:** `/opt/netbird/secrets/kc_admin_password`

The NetBird realm, PKCE client, and management service account are created automatically on first run. No manual Keycloak configuration is required to get the stack running.

## Idempotence

Rerunning the installer:

- preserves generated secrets
- preserves database credentials
- rewrites generated config from the preserved secret set
- refreshes images and restarts the stack

If you intentionally want new credentials, replace the relevant files under `/opt/netbird/secrets` and rerun the script.

## Notes

- The script is self-contained in the repo, but it intentionally writes generated runtime files under `/opt/netbird`
- Reverse proxy configuration remains an operator responsibility
- TURN uses host networking so peers can reach the relay ports directly
- Keycloak runs in production mode (`start`) behind the reverse proxy, with HTTP internally and HTTPS enforced at the proxy layer
